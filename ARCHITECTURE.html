Hunt Pack — Architektur & Regelsprachen

Dokumentation der Regelarchitektur, Regelsprachen und Design-Entscheidungen des HuntingThreats Enterprise Hunt Pack.

✓ Wazuh XML — 500 Regeln ✓ QRadar AQL — 500 Regeln ✓ Google SecOps YARA-L — 500 Regeln ◷ Splunk SPL — geplant

1 Überblick

Das Enterprise Hunt Pack ist eine Sammlung von Threat-Hunting-Regeln für SIEM-Systeme. Jede Regel deckt ein konkretes Angriffsmuster aus dem MITRE ATT&CK Framework ab.

Hunt Pack
├── wazuh/      Wazuh 4.x XML-Regelformat  (500 Regeln ✅ — Batch 1 fertig)
├── qradar/     QRadar AQL-Format           (500 Regeln ✅ — Batch 1 fertig)
├── secops/     Google SecOps YARA-L 2.0   (500 Regeln ✅ — Batch 1 fertig)
└── splunk/     Splunk SPL-Format           (500 Regeln — geplant)

2 Regelsprachen

2.1 Wazuh — XML Rule Language

ElementBedeutung
<rule id="..." level="...">Eindeutige Regel-ID + Severity (1–15)
<if_group>Vorbedingung: Log-Gruppe (z.B. windows)
<field name="..." type="pcre2">Feld-Match gegen PCRE2-Pattern
<mitre><id>MITRE ATT&CK Technik-ID
<frequency> + <timeframe>Threshold-Detektion (N Events in X Sekunden)
<same_field>Korreliert Events über dasselbe Feld
<rule id="100001" level="12">
  <if_group>windows</if_group>
  <field name="win.system.eventID" type="pcre2">^1$|^4688$</field>
  <field name="win.eventdata.commandLine" type="pcre2">(?i)-encodedcommand</field>
  <description>PowerShell: EncodedCommand verwendet (T1059.001)</description>
  <mitre><id>T1059.001</id></mitre>
  <group>execution,powershell,encoded,high_confidence,</group>
</rule>

2.2 Splunk — SPL (geplant)

Splunk-Regeln basieren auf der Splunk Common Information Model (CIM) — normalisierte Feldnamen über alle Log-Quellen.

index=windows source="WinEventLog:Microsoft-Windows-Sysmon/Operational"
  EventCode=1 CommandLine="*-EncodedCommand*"
| eval mitre_technique="T1059.001"
| table _time, host, User, ParentImage, Image, CommandLine

2.3 Google SecOps — YARA-L 2.0

Google SecOps (Chronicle SIEM) nutzt YARA-L 2.0 als native Detektionssprache. Alle Felder werden gegen das UDM (Unified Data Model) normalisiert — plattformunabhängige Feldnamen über alle Log-Quellen.

UDM-Normalisierung: Google SecOps normalisiert alle Logs vor der Regelauswertung. Sysmon Event 1 (Process Create) wird z.B. auf PROCESS_LAUNCH gemappt — Feldname $e.target.process.command_line statt win.eventdata.commandLine.

$e.target.ipZiel-IP-AdresseSysmon Event 3
$e.target.portZiel-PortSysmon Event 3
$e.target.registry.registry_keyRegistry-SchlüsselSysmon Event 13

Minimales Regelformat (YARA-L 2.0)

rule ht_gs_100001_powershell_encoded_command {
  meta:
    rule_id         = "GS-100001"
    author          = "HuntingThreats"
    description     = "PowerShell: -EncodedCommand flag — obfuscated script execution"
    mitre_tactic    = "Execution"
    mitre_technique = "T1059.001"
    severity        = "HIGH"
    confidence      = "HIGH"

  events:
    $e.metadata.event_type = "PROCESS_LAUNCH"
    re.regex($e.target.process.command_line, `(?i)-enc(odedcommand)?\s`) nocase

  condition:
    $e
}

Threshold-Detektion (Beaconing / Spray)

rule ht_gs_104004_beaconing_threshold {
  meta:
    rule_id         = "GS-104004"
    mitre_technique = "T1071.001"
    severity        = "HIGH"

  events:
    $e.metadata.event_type = "NETWORK_CONNECTION"
    NOT $e.target.ip in cidr "10.0.0.0/8"
    NOT $e.target.ip in cidr "192.168.0.0/16"
    NOT $e.target.ip in cidr "172.16.0.0/12"

  match:
    $e.target.ip over 60m        // Gruppiere nach Ziel-IP im 60-Minuten-Fenster

  condition:
    #e > 20                       // Mehr als 20 Verbindungen → Beaconing-Pattern
}

Multi-Event-Korrelation (Attack Chain)

rule ht_gs_110040_initial_access_chain {
  meta:
    rule_id         = "GS-110040"
    mitre_technique = "T1566.001"
    severity        = "CRITICAL"

  events:
    // Event 1: Erfolgreicher externer Login
    $e1.metadata.event_type = "USER_LOGIN"
    NOT $e1.principal.ip in cidr "10.0.0.0/8"
    NOT $e1.principal.ip in cidr "192.168.0.0/16"
    $e1.extensions.auth.auth_details = "Success"

    // Event 2: Recon-Prozess auf gleichem Host
    $e2.metadata.event_type = "PROCESS_LAUNCH"
    re.regex($e2.target.process.file.full_path, `(?i)(whoami|net\.exe|ipconfig)$`)
    $e2.principal.hostname = $e1.target.hostname

    // Event 3: Registry Run-Key schreiben
    $e3.metadata.event_type = "REGISTRY_MODIFICATION"
    re.regex($e3.target.registry.registry_key, `(?i)\\Run$`)
    $e3.principal.hostname = $e1.target.hostname

  match:
    $e1.target.hostname over 30m

  condition:
    $e1 and $e2 and $e3
}

2.4 QRadar — AQL (Ariel Query Language)

Universaler Payload-Ansatz: Da QRadar DSM/CEP-Konfigurationen deploymentspezifisch sind, nutzt das Hunt Pack UTF8(payload) ILIKE '%keyword%' — funktioniert in jedem Standard-Deployment mit WinCollect ohne Custom Event Properties.
ElementBedeutung
SELECT ... FROM eventsFelder die ausgegeben werden
LOGSOURCETYPENAME(logsourceid) ILIKE '%Windows%'Log-Quell-Typ-Filter
UTF8(payload) ILIKE '%muster%'Case-insensitiver Payload-Match (universell)
DATEFORMAT(startTime, 'yyyy-MM-dd HH:mm:ss')Timestamp-Formatierung
LOGSOURCENAME(logsourceid)Hostname / Asset-Name
GROUP BY ... HAVING COUNT(*) >= NThreshold-Detektion
LAST N MINUTESZeitfenster (1440 = 24h Standard)

Standard-Regelformat (AQL)

-- QR-100001 | T1059.001 | PowerShell EncodedCommand
-- Severity: High | Confidence: High
SELECT
  DATEFORMAT(startTime, 'yyyy-MM-dd HH:mm:ss') AS EventTime,
  LOGSOURCENAME(logsourceid) AS LogSource,
  username,
  UTF8(payload) AS EventPayload
FROM events
WHERE (LOGSOURCETYPENAME(logsourceid) ILIKE '%Windows%'
    OR LOGSOURCETYPENAME(logsourceid) ILIKE '%Sysmon%')
  AND (UTF8(payload) ILIKE '%-EncodedCommand%'
    OR UTF8(payload) ILIKE '%-enc %')
LAST 1440 MINUTES;

Threshold-Detektion (AQL)

-- Brute Force: 10+ Fehlversuche von gleicher IP in 15 Minuten
SELECT sourceip, COUNT(*) AS FailCount
FROM events
WHERE LOGSOURCETYPENAME(logsourceid) ILIKE '%Windows%'
  AND UTF8(payload) ILIKE '%EventID>4625<%'
GROUP BY sourceip
HAVING COUNT(*) >= 10
LAST 15 MINUTES;

Private-IP-Ausschluss (Standardmuster)

NOT (destinationip ILIKE '10.%'
  OR destinationip ILIKE '192.168.%'
  OR destinationip ILIKE '172.1%.%'
  OR destinationip ILIKE '127.%')

3.4 Tag-System

-- Wazuh:  <group>credential_access,lsass_access,critical,</group>
-- QRadar: -- Severity: Critical | Confidence: High

Konfidenz-Tags:
  high_confidence  → ~10% FP-Rate oder weniger
  critical         → ~1% FP-Rate (fast ausschließlich bösartig)
  threshold        → Nur mit Frequency / GROUP BY HAVING sinnvoll

4 Designprinzipien

PrinzipUmsetzung
False-Positive-MinimierungParent-Prozess, Pfad und Flags kombinieren; legitime Tools explizit ausschließen
IOSPRegeln beschreiben WAS zu erkennen ist — keine Aktionslogik in Regeln
Keine Panik-FormulierungenNie "Malware gefunden", "infiziert" — stattdessen "Muster erkannt", "Technik detektiert"
MITRE-VollständigkeitJede Regel referenziert mindestens eine Sub-Technik (T1xxx.yyy)
Threshold vor EinzeleventBeaconing, Scanning und Brute-Force immer mit Frequenz-Schwelle
Portable AQL-PatternUTF8(payload) statt Custom Properties — deployment-unabhängig
