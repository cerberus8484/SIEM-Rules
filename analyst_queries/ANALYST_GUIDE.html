01 Was sind Analyst Queries?

Analyst Queries sind fertige, sofort nutzbare Suchabfragen für Security-Analysten im Triage- und Investigation-Alltag. Sie sind interaktiv — du führst sie manuell, gezielt und anlassbezogen aus.

Wann nutze ich eine Analyst Query?

Ein Alert wurde getriggert und du willst mehr Kontext. Ein Ticket läuft: "User X hat seltsames Verhalten gezeigt." Ein Incident ist aktiv und du brauchst schnell eine Timeline. Neue Threat Intelligence: IOC direkt im Environment suchen.

Typische Anwendungsfälle
  • Alert auf Endpunkt → vollständige Host-Timeline aufbauen
  • Verdächtiger Account → alle Aktivitäten in letzten 24h analysieren
  • IOC (IP, Domain, Hash) aus Threat Intel → sofort Environment sweepen
  • Lateral-Movement-Verdacht → SMB/RDP/WMI-Aktivitäten korrelieren
  • Ransomware-Alarm → VSS-Deletion + Service-Stopps prüfen

03 Query-Struktur und Aufbau

Jede Query folgt einem konsistenten Schema — damit du schnell die richtige Query findest und weißt, was du ersetzen musst.

Query-ID Schema

AQ  = Analyst Query (immer gleich)
SPL = Plattform: SPL | QR | GS | WZ
001 = Laufende Nummer (001–050)

AQ-SPL-001  →  Splunk, Query #1
AQ-QR-009   →  QRadar, Query #9
AQ-GS-025   →  Google SecOps, Query #25
AQ-WZ-041   →  Wazuh, Query #41

Annotiertes Beispiel — Splunk

/* AQ-SPL-001 | Triage | Alle Events eines Hosts — letzten 24h
   ↑               ↑       ↑
   Query-ID     Kategorie  Titel

   USE CASE: Erster Überblick nach Alert auf Endpunkt
   ↑ Was beantwortet diese Query?

   PARAMETER: <HOST>
   ↑ Was muss ich ersetzen? */

index=windows host="<HOST>"
| eval EventCodeStr=tostring(EventCode)
| stats count by EventCodeStr, Source
| sort -count
| rename EventCodeStr as "Event Code", count as Anzahl

Annotiertes Beispiel — QRadar

-- AQ-QR-001 | Triage | Alle Events eines Hosts — letzten 24h
   USE CASE: Erster Überblick nach Alert auf Endpunkt
   PARAMETER: <HOST>

SELECT
  DATEFORMAT(devicetime, 'yyyy-MM-dd HH:mm:ss') AS "Zeit",
  category AS "Kategorie",
  eventcount AS "Anzahl"
FROM events
WHERE hostname = '<HOST>'
GROUP BY devicetime, category
ORDER BY devicetime DESC
LAST 1440 MINUTES

Annotiertes Beispiel — Google SecOps UDM

// AQ-GS-001 | Triage | Alle Events eines Hosts — letzten 24h
// USE CASE: Erster Überblick nach Alert auf Endpunkt
// PARAMETER: <HOST>

principal.hostname = "<HOST>"

Annotiertes Beispiel — Wazuh KQL

/* AQ-WZ-001 | Triage | Alle Alerts eines Hosts — letzten 24h
   USE CASE: Erster Überblick nach Alert auf Endpunkt
   PARAMETER: <HOST> */

agent.name: "<HOST>"

06 Verwendung — Splunk

Splunk SPL analyst_queries/splunk/analyst_queries.spl

Wo eingeben?

Splunk Web UI → Search & Reporting → Suchleiste oben

Schritt-für-Schritt

  1. analyst_queries.spl in einem Editor öffnen
  2. Gewünschte Query anhand des Kommentars /* AQ-SPL-XXX */ finden
  3. Alle <PARAMETER> durch echte Werte ersetzen
  4. Nur den Query-Block kopieren (ohne den Kommentar)
  5. In die Splunk-Suchleiste einfügen
  6. Zeitfenster über den Time-Picker oben rechts setzen
  7. Suche starten

Zeitfenster-Optionen

-- Über UI:
Time Picker → "Last 24 hours" / "Last 30 days" / "Custom time"

-- Inline im Query (relativ):
earliest="-24h" latest=now

-- Inline im Query (absolut):
earliest="2026-06-07T08:00:00" latest="2026-06-07T18:00:00"

Nützliche Ergänzungen

-- Mehr Felder sehen:
| table _time, host, user, Image, CommandLine

-- Ergebnisse limitieren:
| head 100

-- Mehrere Werte (OR):
EventCode IN (4624, 4625, 4648)

-- Negation:
NOT Image="*\\svchost.exe"

-- Zeitstempel leserlich:
| eval ts=strftime(_time, "%Y-%m-%d %H:%M:%S")

07 Verwendung — QRadar

QRadar AQL analyst_queries/qradar/analyst_queries.aql

Wo eingeben?

QRadar Console → Log Activity → Add Filter → Advanced Search

Schritt-für-Schritt

  1. analyst_queries.aql in einem Editor öffnen
  2. Query anhand des Kommentars -- AQ-QR-XXX finden
  3. <PARAMETER> durch echte Werte ersetzen
  4. Nur den SELECT-Block kopieren (ohne Kommentare)
  5. In das AQL-Suchfeld einfügen
  6. "Search" oder "Execute Query" klicken

AQL Zeitfenster

-- Relativ (ans Ende der Query):
LAST 60 MINUTES       -- letzte Stunde
LAST 1440 MINUTES     -- letzte 24h
LAST 10080 MINUTES    -- letzte 7 Tage

-- Absolut:
START '2026-06-07 08:00' STOP '2026-06-07 18:00'

Wichtige AQL-Syntax

-- Payload-Suche (case-insensitive):
UTF8(payload) ILIKE '%powershell%'

-- Werte-Liste:
category IN (5001, 5002)

-- Aggregation mit Schwellenwert:
GROUP BY hostname
HAVING COUNT(*) > 10
Tipp

Wenn strukturierte Felder wie hostname leer sind, immer auf UTF8(payload) ILIKE '%...' zurückgreifen — das sucht direkt im Roh-Log.

08 Verwendung — Google SecOps

Google SecOps UDM analyst_queries/secops/analyst_queries.udm

Wo eingeben?

Google SecOps (Chronicle) → Search → UDM Search

UDM Search vs. YARA-L

Diese Queries nutzen die interaktive UDM Search-Oberfläche — nicht YARA-L. YARA-L ist für Detection Rules. UDM Search ist das Analyst-Interface.

Schritt-für-Schritt

  1. analyst_queries.udm in einem Editor öffnen
  2. Query anhand des Kommentars // AQ-GS-XXX finden
  3. <PARAMETER> durch echte Werte ersetzen
  4. Nur den Query-Ausdruck kopieren (ohne Kommentare)
  5. In die UDM Search Bar einfügen
  6. Zeitfenster über Time-Picker setzen
  7. "Search" klicken

09 Verwendung — Wazuh

Wazuh KQL analyst_queries/wazuh/analyst_queries.kql

Wo eingeben?

Wazuh Dashboard → Security Events → KQL-Suchleiste

oder: Discover → Index wazuh-alerts-* → KQL-Suchleiste

Schritt-für-Schritt

  1. analyst_queries.kql in einem Editor öffnen
  2. Query anhand des Kommentars /* AQ-WZ-XXX */ finden
  3. <PARAMETER> durch echte Werte ersetzen
  4. Nur den KQL-Ausdruck kopieren (ohne Kommentar)
  5. In die KQL-Suchleiste einfügen und Enter drücken
  6. Zeitfenster über den Kalender oben rechts setzen

KQL-Syntax-Referenz

-- Gleichheit:
agent.name: "WIN10-CLIENT01"

-- Wildcard-Suche:
data.win.eventdata.commandLine: *powershell*

-- Mehrere Werte (OR):
data.win.system.eventID: ("4624" OR "4625")

-- Numerischer Vergleich:
rule.level >= 10

-- UND-Verknüpfung:
agent.name: "HOST" AND rule.level >= 10

-- Negation:
NOT data.srcip: 10.*

Wichtige Wazuh-Felder

agent.name                         → Hostname
rule.id                            → Wazuh-Rule-ID
rule.level                         → Schweregrad (1–15)
rule.mitre.id                      → ATT&CK Technik-ID
rule.mitre.tactic                  → ATT&CK Taktik

data.win.system.eventID            → Windows Event ID
data.win.eventdata.commandLine     → Commandline
data.win.eventdata.image           → Prozess-Pfad
data.win.eventdata.parentImage     → Elternprozess-Pfad
data.win.eventdata.destinationIp   → Ziel-IP
data.win.eventdata.queryName       → DNS-Query

syscheck.path                      → FIM: überwachter Pfad
syscheck.event                     → added / modified / deleted
Tipp: Immer Agent-Filter zuerst

Setze immer agent.name: "<HOST>" als ersten Filter — sonst sucht Wazuh über alle Agents und erzeugt unnötiges Rauschen.

11 Analyst-Tipps

Die 5 häufigsten Analyst-Fehler

Fehler 1: Zu breites Zeitfenster

Performance-Probleme und zu viele Ergebnisse. Immer mit engem Fenster starten, dann ausweiten.

Fehler 2: Vergessener Host-Filter

Ohne host= / agent.name: sucht man über alle Systeme → Rauschen und langsame Queries.

Fehler 3: Wildcard zu früh

*powershell* matcht auch C:\tools\anti-powershell\uninstall.exe — spezifischer suchen.

Fehler 4: Nur auf Alerts schauen

Manchmal steckt das Interessante in Events ohne Alert-Level. Raw-Logs nicht ignorieren.

Fehler 5: Zeitzone ignorieren

Alert-Timestamp UTC, SIEM in lokaler Zeit → Vorsicht bei der Zeitkorrelation.
